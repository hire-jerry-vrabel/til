## The Setup

I added [Giscus](https://giscus.app) to this blog for reactions
and comments. Giscus is backed by GitHub Discussions — no database,
no backend, just a GitHub repo. It renders inside an iframe and
accepts a `theme` prop to switch between light and dark.

The problem showed up immediately when I toggled dark mode.
The blog switched themes. Giscus stayed dark.

## Why It Broke

The original `useDarkMode` hook stored theme state locally:
```typescript
export function useDarkMode() {
  const [theme, setTheme] = useState<Theme>(() => {
    return localStorage.getItem("theme") as Theme ?? "light"
  })
  // ...
}
```

Every component that called `useDarkMode()` got its own
independent state. The `DarkModeToggle` in the header toggled
its own instance. The `PostComments` component read its own
instance — which never changed because nobody told it to.

Two components reading the same localStorage key but living
in completely separate React state trees. Classic isolation bug.

## The Fix — Lift State into Context

The solution is to lift theme state into a React context so
every component shares the same instance:
```typescript
const ThemeContext = createContext<ThemeContextValue | null>(null)

export function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<Theme>(() => {
    const stored = localStorage.getItem("theme") as Theme | null
    if (stored) return stored
    return window.matchMedia("(prefers-color-scheme: dark)").matches
      ? "dark"
      : "light"
  })

  useEffect(() => {
    document.documentElement.setAttribute("data-theme", theme)
    localStorage.setItem("theme", theme)
  }, [theme])

  const toggle = () => setTheme(t => t === "light" ? "dark" : "light")

  return (
    <ThemeContext.Provider value={{ theme, isDark: theme === "dark", toggle }}>
      {children}
    </ThemeContext.Provider>
  )
}

export function useTheme() {
  const ctx = useContext(ThemeContext)
  if (!ctx) throw new Error("useTheme must be used within ThemeProvider")
  return ctx
}
```

Wrap the app in `ThemeProvider` at the root and every component
that calls `useTheme()` now shares the same reactive state.
When the toggle fires, every subscriber re-renders.

## The Giscus Remount Trick

There's one more wrinkle. Giscus renders inside an iframe —
it doesn't respond to prop changes after initial mount.
Passing a new `theme` prop does nothing once the iframe is loaded.

The fix is to force a full remount when the theme changes
using React's `key` prop:
```typescript
<Giscus
  key={`${slug}-${theme}`}  // remounts when theme changes
  theme={isDark ? "dark" : "light"}
  // ... other props
/>
```

When `key` changes React unmounts and remounts the component
from scratch — which causes Giscus to reload its iframe with
the correct theme. There's a brief flicker on toggle but it's
barely noticeable and worth the correct behavior.

## The Lesson

Two things worth remembering:

**Custom hooks don't share state.** Every call to a hook gets
its own isolated state. If you need multiple components to
share the same reactive value, lift it into context.

**iframes don't respond to prop changes.** If a third-party
component renders in an iframe, the only reliable way to
change its configuration after mount is to force a remount
with a `key` change.

<Callout type="tip">
  The `key` remount trick works for any component that doesn't
  properly respond to prop changes — not just Giscus. It's a
  useful escape hatch when you can't control the component's
  internals.
</Callout>
